# 项目回归测试文档

## 1. 目标

本文件用于在每次代码改动后，快速确认当前项目的核心功能仍可正常运行，避免“改了一个地方，其他功能被带坏”的情况。

本项目当前的测试重点不是单元测试，而是**前端 + 后端 + MATLAB Runtime 联调回归**。

---

## 2. 适用范围

本测试文档覆盖以下内容：

- 前端页面：`webapp/frontend`
- 后端接口：`webapp/backend`
- MATLAB Runtime 联调能力
- 设计点、非设计点、过渡态三类核心计算流程
- 项目文件保存 / 打开 / 导出流程

---

## 3. 环境前提

执行测试前，请先确认：

- 操作系统：Windows
- Python：`3.10`
- Node.js / npm：已安装
- MATLAB Runtime：`R2022b`
- MATLAB Runtime 路径存在：`C:\Program Files\MATLAB\MATLAB Runtime\R2022b`
- 前端依赖已安装
- 后端依赖已安装

后端依赖文件：`webapp/backend/requirements.txt`

前端常用命令位于：

- `webapp/start_webapp.ps1`
- `webapp/run_backend.ps1`
- `webapp/run_frontend.ps1`

MATLAB Runtime 独立运行脚本：

- `run_main_ss.ps1`
- `run_main_ds.ps1`

---

## 4. 重要说明

### 4.1 测试会改写运行产物

执行设计点、非设计点、过渡态计算后，项目中的以下文件可能被改写：

- `MATLAB程序源代码/parameter/*.mat`
- `MATLAB程序源代码/result/RESULT.mat`

因此每次测试后都要检查：

- 当前变更是否只是测试运行产物
- 是否有不应该提交的 `.mat` 文件被修改

### 4.2 当前自动化校验能力有限

当前项目中，现成的自动化检查主要是前端：

```powershell
cd .\webapp\frontend
npm run lint
npm run build
```

后端和 MATLAB Runtime 部分目前仍以**联调冒烟测试 + 关键流程回归**为主。

---

## 5. 测试分级

## 5.1 每次改动后必做

无论改动大小，至少执行以下检查：

- [ ] 前端 `lint` 通过
- [ ] 前端 `build` 通过
- [ ] 前后端可以正常启动
- [ ] `health` 接口可用
- [ ] 至少验证本次改动涉及的页面或流程
- [ ] 检查是否引入非预期的 `.mat` 运行产物修改

## 5.2 合并前 / 大改动后必做

以下情况建议执行完整回归：

- 修改了 `App.tsx`、路由、状态流、公共组件、样式系统
- 修改了后端接口、worker、MAT 文件读写逻辑
- 修改了设计点 / 非设计点 / 过渡态核心计算链路
- 修改了项目文件导入导出逻辑

完整回归包括：

- [ ] 启动与接口联通
- [ ] 结构选型页检查
- [ ] 设计点计算
- [ ] 非设计点计算
- [ ] 过渡态计算
- [ ] 项目文件保存 / 打开 / 另存为
- [ ] CSV / JSON 导出检查
- [ ] 模型自适应页状态检查

## 6. 标准测试启动方式

## 6.1 前端静态检查

```powershell
cd .\webapp\frontend
npm run lint
npm run build
```

**通过标准：**

- `npm run lint` 无报错
- `npm run build` 成功生成生产构建

## 6.2 启动整套 Web 应用

```powershell
powershell -ExecutionPolicy Bypass -File .\webapp\start_webapp.ps1
```

默认行为：

- 启动后端：`127.0.0.1:8000`
- 启动前端：`127.0.0.1:5173`
- 自动打开浏览器

## 6.3 分开启动前后端

### 启动后端

```powershell
powershell -ExecutionPolicy Bypass -File .\webapp\run_backend.ps1
```

### 启动前端

```powershell
powershell -ExecutionPolicy Bypass -File .\webapp\run_frontend.ps1
```

## 6.4 接口冒烟检查

浏览器或接口工具访问：

- `http://127.0.0.1:8000/api/health`
- `http://127.0.0.1:8000/api/defaults`

**期望结果：**

### `GET /api/health`

- 返回成功
- `ok = true`
- `runtimeInstalled = true`（若本机已装 MATLAB Runtime）

### `GET /api/defaults`

- 返回成功
- 含 `topologies`
- 含 `defaults.designPoint`
- 含 `defaults.steadyState`
- 含 `defaults.transient`

## 8. 按改动类型选择测试范围

| 改动类型 | 至少执行的测试 |
| --- | --- |
| 仅前端样式 / 布局调整 | 前端 `lint`、`build`、页面启动、受影响页面冒烟 |
| 前端状态管理 / 组件逻辑修改 | 启动联通 + 受影响页面 + 项目保存/打开 |
| 导入导出功能修改 | 启动联通 + 对应导入导出用例 + 项目文件用例 |

| MATLAB worker / 参数结构修改 | 设计点 + 非设计点 + 过渡态完整回归 |
| 公共入口或主页面修改 | 全量回归 |

## 9. 通过标准

满足以下条件，才可认为本次改动通过回归：

- 前端 `lint` 通过
- 前端 `build` 通过
- 前后端可启动
- 至少本次改动涉及的主流程可正常执行
- 未出现新的白屏、卡死、明显报错、错误导出或错误数据展示
- 未引入未确认的 `.mat` 运行产物提交风险

如果本次改动影响核心链路（设计点 / 非设计点 / 过渡态 / 项目文件），则应执行完整回归。

## 11. 建议执行顺序

如果时间有限，建议每次改动后按下面顺序执行：

1. 前端 `lint`
2. 前端 `build`
3. 启动整套 Web 应用
4. 检查 `health` 和默认参数加载
5. 验证本次改动直接影响的模块
6. 若改动涉及公共逻辑，再补做设计点 / 非设计点 / 过渡态回归
7. 最后检查 `.mat` 运行产物和导出文件是否符合预期
